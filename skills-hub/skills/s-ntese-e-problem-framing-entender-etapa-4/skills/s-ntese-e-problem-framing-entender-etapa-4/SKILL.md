# Skill: Síntese e Problem Framing
**Etapa:** Entender — 1.4
**Objetivo:** Sintetizar tudo que foi descoberto nas etapas anteriores em um problem frame sólido e inegociável — com a dor principal declarada, o impacto mapeado e o porquê da priorização justificado. Nada deve ser assumido: só avança para o Imaginar quem tem o problema muito bem definido.

---

## Seu papel aqui

Conduza ativamente a síntese. Se perceber que o material já está bem organizado, adapte seu papel para estruturar e formalizar — não repita trabalho que já foi feito. Se perceber lacunas, aponte explicitamente antes de avançar.

**Princípio central:** Este é o portão de entrada para o Imaginar. Um problem frame fraco gera soluções para o problema errado. Não deixe passar nada vago, assumido ou sem evidência.

---

## Frameworks que guiam esta etapa

### 1. Affinity Mapping + Clustering Temático
Sintetizar os dados brutos vindos do Discovery (1.2) e do Recorte de Usuário (1.3) em padrões temáticos. Agrupe por afinidade — deixe os clusters emergirem dos dados, não imponha categorias previamente.

**Referências:**
- [User Interviews — Affinity Mapping in 5 Steps](https://www.userinterviews.com/blog/affinity-mapping-ux-research-data-synthesis)
- [DECODE — Affinity Mapping in Product Discovery](https://decode.agency/article/product-discovery-affinity-mapping/)

### 2. Matriz de Dores
Priorizar as dores identificadas em duas dimensões:
- **Frequência** — com que frequência essa dor acontece para o usuário?
- **Intensidade** — qual o impacto quando acontece?

As dores no quadrante de alta frequência + alta intensidade são as candidatas prioritárias ao problem statement.

### 3. POV Statement (Point of View)
Articular o problema do ponto de vista do usuário, combinando três elementos:

*"[Usuário descritivo] precisa [necessidade — verbo] porque [insight revelador]."*

O insight é o que transforma o problema de óbvio em revelador. Se o POV não traz um insight real, ele ainda não está pronto.

**Referências:**
- [IxDF — Define and Frame Your Design Challenge](https://www.interaction-design.org/literature/article/define-and-frame-your-design-challenge-by-creating-your-point-of-view-and-ask-how-might-we)
- [NN/G — User Need Statements](https://www.nngroup.com/articles/user-need-statements/)

### 4. Problem Statement
Formalizar o problema em um enunciado claro, centrado no usuário, que descreve contexto + impacto — sem prescrever solução.

Critérios de um problem statement forte:
- Claro e conciso
- Centrado no usuário
- Descreve o problema, não a solução
- Define o que significa resolver (critério de sucesso)
- Conecta a uma métrica de negócio

**Referências:**
- [NN/G — Problem Statements in UX Discovery](https://www.nngroup.com/articles/problem-statements/)
- [ProdPad — The Product Management Problem Statement](https://www.prodpad.com/blog/product-management-problem-statement/)

### 5. Double Diamond — Define Stage
Checklist de prontidão antes de avançar para o Imaginar:
- Insights coletados foram sintetizados e interpretados
- Problema foi redefinido (não apenas confirmado)
- User need statement foi articulado
- A dor prioritária está justificada com dados
- O problem frame está alinhado com a métrica de negócio

**Referência:** [Design Council — The Double Diamond](https://www.designcouncil.org.uk/our-resources/the-double-diamond/)

## Fluxo de raciocínio

1. **Perguntas iniciais** — entenda o que vem das etapas anteriores:
   - Qual é a lista de dores brutas levantada no Discovery?
   - Qual é o recorte de usuário foco definido na 1.3?
   - Existe material já sintetizado ou tudo ainda está bruto?

2. **Affinity Mapping + Clustering** — organize os dados brutos em clusters temáticos se ainda não foi feito

3. **Matriz de Dores** — priorize as dores por frequência x intensidade. Identifique a dor principal

4. **POV Statement** — articule o problema do ponto de vista do usuário com o insight revelador

5. **Problem Statement** — formalize o problema: contexto + impacto + critério de sucesso + conexão com métrica de negócio

6. **Checklist de prontidão** — valide com os critérios do Double Diamond antes de declarar encerrada a etapa

---

## Como interagir

- Se o material estiver bruto, conduza ativamente a síntese
- Se o material já estiver organizado, estruture e formalize — não refaça o que já está feito
- Aponte lacunas explicitamente: se algo não está embasado em dados, sinalize antes de avançar
- Não deixe passar nada vago — o problem frame precisa estar inegociável antes de liberar para o Imaginar
- Nunca invente dados ou presuma informações que não foram trazidas

---

## Critério de avanço

Só libere para o **Imaginar** quando estiver claro:
- Qual é a **dor principal** que queremos resolver
- **Por que essa dor foi priorizada** (dados + raciocínio)
- **Quem é o usuário** que sofre essa dor
- **O que isso impacta** — para o usuário e para o negócio
- **Como saberemos que resolvemos** — métrica de sucesso
